# BeanVault 豆仓 — 产品需求文档 (PRD)

---

## Problem Statement

精品咖啡爱好者通常会同时持有 3-8 包不同产区、不同烘焙度的咖啡豆。每包豆子都有独特的**养豆期**（烘焙后需要静置 X 天才能达到最佳风味），错过最佳饮用窗口是常态。现有的笔记 App 或表格工具无法自动计算养豆完成时间，也无法按"可以喝了"排序，导致用户要么提前开封浪费豆子，要么忘记某包豆子已经养好。

用户需要一个**以养豆期自动计算为核心**的轻量级咖啡豆库存管理工具，支持离线使用、数据云端同步，且操作路径最短。

## Solution

BeanVault 是一个移动端 PWA 应用，核心能力：

1. **养豆期自动计算**：根据生产日期 + 养豆天数自动判定"已养好 / 还需 X 天"，并在卡片上绿色/橙色标识
2. **极速筛选与搜索**：状态 tab（全部/正在喝/架子上/冰箱/已喝完/回收站）+ 分类标签 + 产国标签 + 全文搜索，所有筛选叠加即时生效
3. **半自动状态流转**：养好后卡片显示"开始喝"快捷按钮，用户一键确认切换状态，兼顾自动计算与手动控制
4. **离线可用 + 云端同步**：IndexedDB 本地存储保证离线读写，Supabase 云端同步保证换设备不丢失
5. **类原生移动体验**：PWA 添加到桌面，全屏启动，适配安全区域，触控友好

## User Stories

1. 作为一名咖啡爱好者，我想要在手机上快速录入一包新豆子的信息（名称、产地、烘焙度、养豆天数、生产日期等），以便追踪我的每一包库存
2. 作为一名咖啡爱好者，我想要看到每包豆子的养豆倒计时（已养好/还需 X 天），以便我知道哪包豆子现在可以喝了
3. 作为一名咖啡爱好者，我想要在豆子养好后一键标记为"正在喝"，以便系统自动将状态从架子上切换到正在喝
4. 作为一名咖啡爱好者，我想要在喝完一包豆子后标记为"已喝完"，以便它从活跃库存中移出但仍可追溯
5. 作为一名咖啡爱好者，我想要按状态筛选豆子（全部/正在喝/架子上/冰箱/已喝完/回收站），以便快速找到当前关注的豆子
6. 作为一名咖啡爱好者，我想要按分类（手冲/意式/订阅卡）和产国快速筛选豆子，以便在多个维度下缩小范围
7. 作为一名咖啡爱好者，我想要通过关键词搜索豆子名称、庄园、产区、风味描述，以便快速定位某包特定的豆子
8. 作为一名咖啡爱好者，我想要切换排序方式（生产日期倒序/可以喝了排序），以便按我的当前需求查看列表
9. 作为一名咖啡爱好者，我想要在离线状态下正常查看和编辑我的豆子库存，以便在没有网络时不受影响
10. 作为一名咖啡爱好者，我想要注册/登录后自动将数据同步到云端，以便换手机或重装 App 后数据不丢失
11. 作为一名咖啡爱好者，我想要通过邮箱接收验证码的方式登录，无需记住密码，以便快速安全地完成认证
12. 作为一名咖啡爱好者，我想要导出我的所有豆子数据为 JSON 文件，以便备份或分享给其他咖啡同好
13. 作为一名咖啡爱好者，我想要从 JSON 或 CSV 文件导入豆子数据，以便批量迁移历史记录或从其他工具切换过来
14. 作为一名咖啡爱好者，我想要在导入数据时选择合并或替换，以便根据场景灵活处理已有数据
15. 作为一名咖啡爱好者，我想要在误删豆子后从回收站恢复，以便避免操作失误导致数据永久丢失
16. 作为一名咖啡爱好者，我想要在首次打开 App 时看到引导性的空状态页面，以便理解这个 App 能为我做什么
17. 作为一名咖啡爱好者，我想要在添加/编辑/删除操作后看到 Toast 确认反馈，以便确认操作已生效
18. 作为一名咖啡爱好者，我想要在执行删除或标记已喝完等操作时看到确认弹窗，以便避免误操作
19. 作为一名咖啡爱好者，我想要在离线时看到顶部横幅提示"当前离线，数据仅保存在本地"，以便了解当前数据同步状态
20. 作为一名咖啡爱好者，我想要将 App 添加到手机桌面并以全屏方式打开，以便获得接近原生 App 的使用体验
21. 作为一名咖啡爱好者，我想要在录入豆子时从预置的产国列表中选择（带国旗 emoji），也可自定义输入，以便快速准确地标注产地
22. 作为一名咖啡爱好者，我想要在录入风味描述时看到预置的风味词库建议，同时也支持自由输入，以便快速且一致地标注风味
23. 作为一名咖啡爱好者，我想要在豆子卡片上看到关键信息概览（名称、产地国旗、庄园、风味标签、状态、养豆状态、单价、生产日期），以便在列表页就能掌握每包豆子的核心信息
24. 作为一名咖啡爱好者，我想要在卡片上直接点击"开始喝"或"喝完了"图标按钮，以便无需进入详情页即可完成高频操作
25. 作为一名咖啡爱好者，我想要查看和编辑每包豆子的完整信息，以便在详情页中修改任何字段
26. 作为一名咖啡爱好者，我想要我的排序偏好（生产日期/可以喝了）被记住，以便下次打开 App 时延续我的使用习惯

## Implementation Decisions

### 技术栈

- **前端框架**: React 18+ with TypeScript
- **构建工具**: Vite 5+
- **路由**: React Router v7
- **状态管理**: Zustand（persist 中间件对接 localforage/IndexedDB）
- **UI 框架**: Tailwind CSS（design token 映射自 DESIGN.md）+ Radix UI（无样式交互原语）
- **PWA**: vite-plugin-pwa（基于 Workbox）
- **后端/数据库**: Supabase（Auth + PostgreSQL + Row Level Security）
- **认证方式**: 邮箱 OTP 验证码（Magic Link 的 OTP 模式，避免 PWA 跳转问题）

### 数据模型

```typescript
interface Bean {
  id: string; // uuid
  name: string;
  category: "pourover" | "espresso" | "subscription";
  status: "shelf" | "fridge" | "drinking" | "finished";
  country: string; // 产国全称
  countryCode?: string; // ISO 代码，用于国旗映射
  estate: string;
  variety: string;
  process: string;
  roastLevel: string;
  flavorNotes: string[]; // 风味标签数组
  pricePerGram: number; // 元/克
  restingDays: number; // 养豆天数
  productionDate: string; // "YYYY-MM-DD"
  isDeleted: boolean; // 软删除标记
  createdAt: string; // ISO 8601
  updatedAt: string; // ISO 8601
}
```

### 计算属性（前端派生，不存储）

- `restedDate`: `productionDate + restingDays` 的日期对象
- `isRested`: 当前日期 >= restedDate
- `daysUntilRested`: `isRested ? 0 : 差值天数`

### 路由结构

| 路由 | 页面 | 说明 |
| --- | --- | --- |
| `/` | 豆仓首页 | 库存列表 + 筛选栏 + 搜索 + 悬浮添加按钮 |
| `/bean/:id` | 豆子详情 | 查看/编辑豆子信息，标记已喝完，删除 |
| `/add` | 添加豆子 | 录入新豆子表单 |
| `/settings` | 设置 | 导入/导出、云同步、回收站入口、关于 |
| `/auth/callback` | 认证回调 | Supabase OTP 验证码确认 |

### 底部导航

- 2 项：豆仓 + 设置
- 添加豆子通过首页右下角悬浮 + 按钮
- 搜索在首页顶部固定显示

### 状态筛选逻辑

首页顶部 6 个 tab，均为纯状态值筛选：

| Tab | 筛选条件 |
| --- | --- |
| 全部 | 所有非软删豆子 |
| 正在喝 | `status === 'drinking'` |
| 架子上 | `status === 'shelf'` |
| 冰箱 | `status === 'fridge'` |
| 已喝完 | `status === 'finished'` |
| 回收站 | `isDeleted === true` |

### 养豆期计算规则

- 养豆完成日期 = 生产日期 + 养豆天数
- 卡片展示：已养好 → 绿色圆点 + "养好了"；养豆中 → 橙色圆点 + "还需 X 天"
- 当 `isRested === true` 且 `status === 'shelf'` 时，卡片显示"开始喝"快捷按钮
- 当 `status === 'drinking'` 时，卡片显示"喝完了"快捷按钮
- "可以喝了"排序：已养好的排最前（内部按生产日期倒序），养豆中的按还需天数升序

### 搜索与筛选

- 即时搜索（无防抖），匹配范围：name, estate, country, variety, flavorNotes 拼接
- 分类标签：全部 / 手冲 / 意式 / 订阅卡（单选）
- 产国标签：动态生成已出现产国 + 国旗 emoji（单选）
- 所有筛选条件 AND 叠加，立即更新列表

### 表单字段规范

| 字段 | 类型 | 必填 | 选项/约束 |
| --- | --- | --- | --- |
| 豆子名称 | 文本 | ✅ | 自由输入 |
| 分类 | 单选按钮组 | ✅ | 手冲 / 意式 / 订阅卡 |
| 状态 | 单选按钮组 | ✅ | 架子上 / 冰箱 / 正在喝（新豆子默认架子上） |
| 产国 | 下拉选择+国旗 | ✅ | 预置 30-50 常见产国，支持自定义输入 |
| 庄园 | 文本 | 否 | 自由输入 |
| 豆种 | 文本 | 否 | 自由输入 |
| 处理法 | 单选 | 否 | 日晒 / 水洗 / 蜜处理 / 厌氧发酵 / 其他（自定义） |
| 烘焙度 | 单选 | 否 | 极浅烘 / 浅烘 / 中烘 / 深烘 |
| 风味描述 | 标签输入 | 否 | 预置 20-30 词风味词库建议 + 逗号/回车生成标签 + 自由输入 |
| 克单价 | 数字输入 | 否 | 元/克，保留两位小数 |
| 养豆天数 | 数字步进 | ✅ | 默认 14，最小 0，非负整数 |
| 生产日期 | 日期选择器 | ✅ | 默认当天，不可晚于今天 |

### 删除策略

- 本地和云端统一软删（`isDeleted: true`）
- 回收站作为首页第 6 个 tab，展示已删除豆子，支持恢复或彻底删除
- 删除操作需居中 AlertDialog 确认

### 导入导出

- **导出**: 所有豆子导出为 JSON 文件 `{ "version": 1, "beans": [...] }`
- **导入**: 支持 JSON 和 CSV 格式
- **导入策略**: 弹窗二选一 —— 合并（同 id 覆盖，新 id 追加）或 全部替换（清空现有再导入）
- CSV 导入时校验必填字段，缺失则跳过并提示行号

### 云端同步

- 使用 Supabase 作为后端
- 同步策略：以 `updatedAt` 最新为准进行双向同步
- 同步时机：启动时（若已登录且在线）+ 每次增删改操作后乐观推送
- 离线时所有操作仅落在本地 IndexedDB，网络恢复后自动重试同步
- 离线时顶部显示黄色横幅"当前离线，数据仅保存在本地"

### PWA 配置

- `manifest.json`: `display: standalone`，图标 192x192 + 512x512
- Service Worker（vite-plugin-pwa + Workbox）：App Shell 预缓存，静态资源 CacheFirst，API NetworkFirst
- 安全区域适配：`env(safe-area-inset-*)`
- 触控区域 ≥ 44x44pt
- 支持 iOS Safari 15+，Android Chrome 90+

### 交互反馈

- 成功/失败操作均显示 Toast 通知（顶部或底部滑出，2-3 秒自动消失）
- 不可逆操作（删除、标记已喝完）需居中 AlertDialog 确认
- 排序偏好通过 Zustand persist 持久化
- 首次无数据时展示空状态插画 + 引导文案 + 醒目的添加按钮

### 产国列表

前端硬编码 30-50 个常见咖啡产国，每个含国家名称、ISO 代码、国旗 emoji（通过 `String.fromCodePoint` 生成）。支持自定义输入兜底。

### 风味词库

预置 20-30 个 SCA 风味轮常见词汇作为输入建议，支持自由输入新词。

## Testing Decisions

### 测试范围

由于项目当前无已有测试框架，MVP 阶段测试策略如下：

- **单元测试**（Vitest）：Zustand store 的 reducer 逻辑（筛选、排序、养豆计算）、Bean 数据校验函数
- **组件测试**（React Testing Library）：关键交互组件（BeanCard、FilterTabs、AddForm）的用户行为验证
- **E2E 测试**（Playwright，可选）：核心用户流程（添加豆子 → 养豆倒计时 → 标记开始喝 → 标记已喝完 → 删除恢复）

### 测试原则

- 只测试外部行为，不测试实现细节
- 养豆期计算是核心逻辑，需覆盖边界情况（跨年、闰年、养豆天数 0）
- 筛选叠加逻辑需覆盖各种组合
- 同步冲突以 `updatedAt` 比较为核心的逻辑需覆盖

### 测试 Seam

- 最高 seam：Zustand store 的 action 和 selector —— 这是 UI 和数据之间的契约层
- 中间 seam：localforage 的读写接口 —— 可以 mock 以测试离线行为
- 底层 seam：Supabase client 的 query/mutation —— 可以 mock 以测试同步逻辑

## Out of Scope

以下功能不在 MVP 范围内：

- 订阅卡的独立管理模块（订阅周期、到期提醒、历史记录）
- 多个用户之间的库存共享或社交功能
- 冲煮参数记录（水温、研磨度、粉水比、冲煮时间等）
- 豆子评分或品鉴笔记
- 购买链接或价格追踪
- 暗色模式
- 国际化（仅中文界面）
- Apple 登录 / Google 登录等第三方 OAuth
- 原生 App（iOS/Android）壳
- 推送通知（养豆完成提醒等）
- 图片上传（豆子包装照片）
- 数据统计面板（喝了多少包、哪个产区最多等）
